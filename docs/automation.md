# Способы запуска, автозапуск и авто-failover с Surge

Три способа запускать `vk-turn-socks` — от простого к удобному — плюс
автоматика с Surge, чтобы туннель включался сам, когда прямой интернет
недоступен, и выключался, когда снова доступен.

## Установка приложения (в пару кликов)

1. Скачайте из [Releases](../../releases) файл **`VK-Turn-Proxy-Agent.dmg`**
   (он **универсальный** — работает и на Apple Silicon, и на Intel, выбирать
   архитектуру не нужно).
2. Откройте DMG и перетащите **«VK Turn Proxy Agent»** в **Applications**.
3. Запустите из Applications — иконка появится в строке меню.
4. Первый запуск (сборка не подписана Developer ID): правый клик по приложению
   → **Open** → **Open** ещё раз. Либо один раз в терминале:
   `xattr -dr com.apple.quarantine "/Applications/VK Turn Proxy Agent.app"`.
5. В панели агента нажмите **«Edit config…»**, заполните сервер/ключи
   (см. [docs/config.md](config.md)) и включите туннель.

Только headless-бинарь без UI — там же в Releases: `vk-turn-socks-darwin-arm64.zip`
(Apple Silicon) или `-amd64.zip` (Intel).

> Собрать DMG самому: `scripts/package_dmg.sh` (нужны Go + Xcode CLT + xcodegen).
> Релизы собираются автоматически на GitHub Actions при пуше тега `vX.Y.Z`
> (см. `.github/workflows/release.yml`).

## Где хранит конфиг

Один-единственный файл, общий для приложения, сервиса и CLI:

```
~/Library/Application Support/VKTurnProxy/config.json
```

`vk-turn-socks` без флага `-config` всегда использует именно его (флаг `-config`
оставлен как override для отладки). Открыть/отредактировать его проще всего из
трея: кнопки **«Edit config…»** (откроет в редакторе) и **«Reveal»** (покажет в
Finder) в панели агента.

Быстро создать конфиг из настроек iOS:

```shell
./vk-turn-socks -import 'vkturnproxy://import?data=…'
# файл запишется в ~/Library/Application Support/VKTurnProxy/config.json
```

## Способ 1. Терминал (как сейчас)

```shell
./vk-turn-socks            # берёт ./config.json или app-support конфиг
```

Просто, но окно терминала должно оставаться открытым. Дальше — как этого избежать.

## Способ 2. Фоновый сервис (launchd) — без открытого терминала

Скрипт [`scripts/service.sh`](../scripts/service.sh) ставит пользовательский
LaunchAgent: сервис стартует при входе в систему, перезапускается при падении,
пишет лог в файл. Root не нужен, аккаунт Apple не нужен.

```shell
scripts/service.sh install     # собрать/скопировать бинарь, поставить и запустить
scripts/service.sh status      # состояние
scripts/service.sh logs        # хвост лога
scripts/service.sh stop        # остановить (агент остаётся установленным)
scripts/service.sh start       # запустить снова
scripts/service.sh uninstall   # удалить агент
```

`install` кладёт бинарь и конфиг в `~/Library/Application Support/VKTurnProxy/`
и, если конфига ещё нет, копирует шаблон — заполните его перед стартом
(см. [docs/config.md](config.md)). Логи — `socks.out.log` / `socks.err.log`
в той же папке.

Это чистый «сервис»: включил один раз — работает в фоне и сам поднимается после
перезагрузки. Включать/выключать — `start` / `stop`.

## Способ 3. Менюбар-агент (иконка в трее со статистикой) — рекомендуется

`VK Turn Proxy Agent.app` — маленькое приложение в строке меню. По клику на
иконку открывается панель со **статистикой в реальном времени**: скорость ↑/↓,
суммарный трафик, активные/всего соединения, состояние пула credential'ов,
uptime, IP TURN-релея. Там же — тумблер Connect/Disconnect, переключатель
**Auto (failover)** и кнопка решения капчи, если она понадобится.

Это обычная утилита (`LSUIElement`, без иконки в Dock) — не системное
расширение, аккаунт Apple не требуется. Сборка (нужны Go + Xcode CLT + xcodegen):

```shell
scripts/build_menubar.sh
open "build/DerivedData/Build/Products/Release/VK Turn Proxy Agent.app"
```

Конфиг — тот же файл в `Application Support`; при первом запуске нажмите в
панели «Edit config…».

Иконка отражает состояние: серый щит — выключено, жёлтый — подключается/директ
жив (в Auto), зелёный — подключено и есть трафик, оранжевый с восклицательным
знаком — нужна капча.

## Авто-failover с Surge (DIRECT ⇆ VK Turn)

Задача: пока прямой интернет доступен — ходить `DIRECT`; как только прямой
доступ пропадает — автоматически уходить в туннель; когда прямой доступ
возвращается — снова `DIRECT`. Причём проверять доступность нужно **в обход
туннеля**.

Здесь две независимые части: **маршрутизация** (это делает Surge) и
**жизненный цикл процесса** (по желанию — менюбар-агент в режиме Auto).

### Часть 1. Маршрутизация — нативная `fallback`-группа Surge

Surge умеет ровно это без скриптов — группа типа `fallback` постоянно
health-check'ает политики и выбирает первую живую:

```ini
[Proxy]
VKTurn = socks5, 127.0.0.1, 1080, udp-relay=true

[Proxy Group]
# DIRECT, пока он проходит health-check; иначе — VKTurn. Как только DIRECT
# снова оживает, Surge сам возвращается на него.
Auto = fallback, DIRECT, VKTurn, url=http://www.gstatic.com/generate_204, interval=30, timeout=3

[Rule]
# Пробу связности всегда гоним НАПРЯМУЮ — чтобы health-check DIRECT отражал
# реальный прямой доступ, а не состояние туннеля.
DOMAIN,www.gstatic.com,DIRECT
# Трафик самого сервиса и релея — всегда напрямую (без петли):
PROCESS-NAME,vk-turn-socks,DIRECT
IP-CIDR,95.163.34.140/32,DIRECT
IP-CIDR,95.163.34.145/32,DIRECT
IP-CIDR,127.0.0.1/32,DIRECT
FINAL,Auto
```

- `fallback` проверяет DIRECT по его собственному пути. Пока `generate_204`
  доступен напрямую — весь трафик идёт `DIRECT`. Как только напрямую перестаёт
  открываться — Surge переключает на `VKTurn`. Вернулся прямой доступ — Surge
  вернулся на `DIRECT`. Это и есть требуемое поведение.
- IP релеев берите из лога (`vkcalls: turn_server.urls[...]`).

Если туннель у вас **всегда запущен** (Способ 2 или 3 без Auto) — этого блока
достаточно: Surge сам решает, когда им пользоваться.

### Часть 2. Жизненный цикл — режим Auto в менюбар-агенте (экономия ресурсов)

Держать 30 TURN-аллокаций к VK постоянно необязательно. Включите в агенте
**Auto (failover)** — тогда агент сам:

- каждые ~5 c пробует прямой доступ к тому же `generate_204`;
- если прямой доступ пропал (2 неудачи подряд) — **запускает** туннель
  (и Surge-`fallback` тут же уводит трафик в `VKTurn`);
- если прямой доступ восстановился (3 успеха подряд) — **останавливает**
  туннель (Surge возвращается на `DIRECT`).

Важно: правило `DOMAIN,www.gstatic.com,DIRECT` из части 1 обязательно — оно
заставляет пробу агента идти напрямую даже когда туннель поднят, иначе агент
не увидит восстановление прямого доступа. (Проба хоста жёстко направлена
`DIRECT`, поэтому «в обход TUN» выполняется.)

Итого: Surge отвечает за маршрут, агент — за то, чтобы тяжёлый туннель работал
только когда он реально нужен. Обе части используют один и тот же сигнал —
доступность прямого пути.

> Примечание про Surge enhanced-mode (системный TUN): без правил
> `PROCESS-NAME,vk-turn-socks,DIRECT` и IP релеев `DIRECT` трафик сервиса к
> VK-релею мог бы завернуться обратно в прокси. Правила выше это исключают —
> сервис всегда выходит напрямую.
