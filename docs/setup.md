## Как это работает

Приложение получает IP адрес и порт TURN relay из URL звонка VK (или берёт эту информацию непосредственно из настроек). Звонок необходимо создать вручную и скопировать ссылку в настройки приложения (или найти ссылку через поиск).

Затем приложение строит туннель к TURN relay. С точки зрения самого TURN relay, приложение создаёт набор аллокаций (соединений) от устройства до него и указывает ему IP адрес и порт сервера, к которому уже строит аллокации (соединения) сам TURN relay.

Чтобы принять эти соединения, на сервере должно работать vk-turn-proxy приложение, которое реализует тот же протокол, что и клиентское приложение. В зависимости от выбранного режима работы сетевые пакеты оборачиваются различными дополнительными уровнями шифрования и имитации сетевых протоколов (но во всех режимах присутствует WireGuard слой) и отправляются к TURN relay поверх созданных соединений, который перекидывает их на сервер.

Сервер снимает эти дополнительные уровни (кроме WireGuard) и отдаёт пакеты WireGuard серверу, который в общем случае должен быть запущен рядом на том же сервере. В клиентском приложении WireGuard часть встроена внутри, дополнительно запускать WireGuard на устройстве не надо.

Обратное движение пакетов аналогичное — на сервере WireGuard отдаёт пакеты серверному приложению vk-turn-proxy, оно оборачивает пакеты дополнительными уровнями и отправляет TURN relay, тот отправляет клиентскому приложению.

> Это ровно тот же движок, что и в [iOS-версии](https://github.com/anton48/vk-turn-proxy-ios): ядро на Go (`pkg/proxy`, `pkg/turnbind`, `WireGuardBridge`) скомпилировано в `WireGuardTURN.xcframework` без изменений, изменилась только обёртка вокруг него (см. README).

## Что необходимо для работы

- URL VK звонка (или адрес/порт TURN relay)

- выделенный или VPS сервер, на котором будет работать серверное приложение vk-turn-proxy. К нему будет подключаться TURN relay, поэтому сервер должен иметь публичный IP адрес, достижимый из Интернет, и на firewall (если он есть) должен быть открыт доступ к паре IP:порт.

- на сервере должен быть установлен, настроен и запущен [WireGuard сервер](https://www.wireguard.com) (или другая реализация WireGuard протокола).

- в общем случае, на сервере должна быть настроена маршрутизация между интерфейсами (например, `net.ipv4.ip_forward=1` в Linux или `net.inet.ip.forwarding=1` в FreeBSD).

- в общем случае, на сервере должен быть настроен NAT, который применяется к IP адресам клиентов WireGuard. Другой вариант — когда пакеты клиентов WireGuard отправляются дальше в какой-то другой туннель.

## Режимы работы

Клиентское приложение поддерживает 4 режима работы (выбираются в Settings → Server mode):

### Legacy

Режим совместимости с [оригинальным](https://github.com/cacggghp/vk-turn-proxy) сервером. Продолжает работать, но даёт очень низкую скорость из-за шейпинга на стороне VK (если используется VK TURN relay).

### SRTP

Режим по умолчанию. Требует использования сервера, который понимает данный протокол.

Для запуска серверной части необходимо скачать бинарный файл под нужную ОС и архитектуру из раздела [Releases](https://github.com/anton48/vk-turn-proxy/releases), дать права на исполнение и запускать с ключами:

```shell
./server-linux-amd64 -listen XXX.XXX.XXX.XXX:56000 -connect YYY.YYY.YYY.YYY:51820 -srtp -logfile /var/log/srtp.server.log
```

- `XXX.XXX.XXX.XXX` — IP адрес сервера (в общем случае публичный)
- `56000` — порт, на котором слушает серверное приложение (при наличии firewall разрешите доступ к IP:port)
- `YYY.YYY.YYY.YYY:51820` — адрес и порт WireGuard сервера (IP обычно локальный, 127.0.0.1)

### SRTP-WRAP

Режим совместимости с серверами других авторов:
- https://github.com/Moroka8/vk-turn-proxy/releases
- https://github.com/samosvalishe/vk-turn-proxy/releases

### SRTP-WRAP-A

Режим совместимости с сервером [WDTT](https://github.com/amurcanov/proxy-turn-vk-android/releases). В этом сервере WireGuard встроен внутрь серверного приложения, поэтому отдельно запускать и настраивать его на сервере не надо.

## Настройки клиентского приложения

### Legacy или SRTP режим

В разделе «VK TURN Proxy» вводятся:

- URL VK звонка (`https://vk.ru/call/join/xxxxxxxxxxxxxxxxxxxxxx`)
- IP адрес и порт сервера, где запущена серверная часть vk-turn-proxy (то, что в `-listen XXX.XXX.XXX.XXX:56000`)
- опционально адрес и порт непосредственно TURN relay (в общем случае оставляем пустым)
- режим подключения (Server mode)
- опционально UDP в качестве транспорта (может быть быстрее TCP по умолчанию, но TCP значительно устойчивее)
- количество соединений (10-20-30), от этого зависит общая скорость (на 30 соединениях в идеальных условиях можно получить 50+ Мбит/сек)
- таймаут между попытками при неудачном решении капчи

В разделе WireGuard вводятся:

- приватный ключ клиента (на стороне WireGuard сервера в разделе `[Peer]` этого клиента должен быть парный публичный ключ)
- публичный ключ WireGuard сервера
- опционально preshared key
- IP адрес и маска (соответствуют `AllowedIPs` в разделе `[Peer]` этого клиента на сервере)
- IP адрес DNS сервера поверх туннеля

### SRTP-WRAP режим

Появляется дополнительное поле WRAP-ключа. Ключ должен совпадать с заданным на сервере через `-wrap-key` (64 символа 16-ричного числа, 32 байта). Сгенерировать можно на сервере через `-gen-wrap-key`.

### SRTP-WRAP-A режим

Появляется поле пароля WDTT, а раздел WireGuard исчезает (WDTT сервер присылает настройки автоматически после установления соединения):

- публичный адрес сервера WDTT
- порт (по умолчанию 56000)
- пароль («главный пароль туннеля» в терминах WDTT)

## Автоматические ссылки

Приложение регистрирует ссылки вида `vkturnproxy://` на себя — при клике на такую ссылку macOS передаёт её в приложение. Альтернативно: Settings → «Import from Connection Link…» (читает ссылку из буфера обмена).

Ссылка содержит в закодированном base64 виде настройки, позволяющие заполнить их автоматически. Для создания ссылок прилагается [скрипт](../quick_link.py) на Python (работает так же, как в iOS-версии):

```shell
python3 quick_link.py -gen-peer-key 192.168.100.2/24
```

Дополнительно поддерживается импорт `wdtt://` ссылок из WDTT.

## Backup/restore

Settings → «Export Full Backup…» / «Import Full Backup…». На macOS для сохранения используется системная панель сохранения (NSSavePanel), для импорта — панель открытия файла (NSOpenPanel).

Export сохраняет json-файл со всеми настройками и дополнительно добытыми credentials от VK (их время действия 8 часов, устаревшие значения игнорируются). Import заменяет существующие настройки данными из файла. «Reset TURN Cache» обнуляет закэшированные credentials.

## Капча

По умолчанию приложение использует путь получения VK credentials, который в настоящий момент не требует решения капчи. Если он не срабатывает — задействуется fallback на путь с капчей, которую приложение пытается решить автоматически. Если и это не срабатывает — открывается WebView (встроенный браузер) для ручного решения. WebView доступен в процессе установления соединения.

## Сборка и установка на macOS

### Требования

- Mac с macOS 13.0 (Ventura) или новее
- Xcode 16.2+ с macOS SDK и command-line tools
- Go 1.25.5+ (`brew install go`)
- [`xcodegen`](https://github.com/yonaskolb/XcodeGen) (`brew install xcodegen`)
- Платный аккаунт Apple Developer — capability **Network Extensions** (packet tunnel) недоступна для бесплатного personal team

### Шаги

```shell
# 1) собрать Go-ядро в универсальный (arm64 + x86_64) WireGuardTURN.xcframework
make bridge

# 2) сгенерировать Xcode-проект из VKTurnProxy/project.yml
make project

# 3) открыть и собрать
open VKTurnProxy/VKTurnProxy.xcodeproj
```

Шаг `make bridge` создаёт пропатченную копию GOROOT (замена `mach_absolute_time`
на `mach_continuous_time`, чтобы таймеры Go продолжали тикать во время сна Mac),
компилирует ядро под обе архитектуры и собирает `WireGuardTURN.xcframework`.

### Team ID и идентификаторы

Перед сборкой замените плейсхолдеры своими значениями (везде сейчас стоит
демонстрационный Team ID `CDMQ33VFQC`):

1. `VKTurnProxy/project.yml` → `DEVELOPMENT_TEAM`.
2. `VKTurnProxy/VKTurnProxy/AppGroup.swift` → `identifier` и `keychainAccessGroup`
   (замените префикс `CDMQ33VFQC.` на свой Team ID).
3. В `App ID` вашего аккаунта включите capability:
   - **App Groups** (создайте группу `<TeamID>.com.vkturnproxy.mac` для приложения и расширения)
   - **Network Extensions**
   - **System Extension** (для контейнер-приложения)
   - **Keychain Sharing** (группа `com.vkturnproxy.shared`)

App Group на macOS обязан быть с префиксом Team ID (в отличие от iOS-формы
`group.*`). В entitlements это записано как `$(TeamIdentifierPrefix)com.vkturnproxy.mac`
и раскрывается в `<TeamID>.com.vkturnproxy.mac` — то же значение зашито в
`AppGroup.identifier`. Три места (константа + два entitlements-файла) должны
совпадать байт в байт, иначе приложение и расширение будут работать с разными
контейнерами и перестанут делить `vpn.log` / `creds-pool.json` / `vk_profile.json`.

### Первый запуск: подтверждение системного расширения

На macOS packet-tunnel-провайдер, распространяемый вне Mac App Store, — это
**системное расширение** (Apple TN3134). При первом запуске приложение попросит
разрешение:

1. Запустите `VK Turn Proxy.app` (для установки системных расширений приложение
   должно лежать в `/Applications`).
2. macOS покажет запрос на разрешение расширения. Откройте **Системные настройки →
   Основные → Объекты входа и расширения → Сетевые расширения** (на Ventura:
   **Конфиденциальность и безопасность**) и разрешите «VK Turn Proxy».
3. После одобрения баннер в приложении исчезнет, и кнопка Connect станет активной.
4. При первом подключении macOS также попросит разрешить добавление VPN-конфигурации.

Прогресс установки/одобрения расширения отражается баннером на главном экране
(этой логики нет в iOS-версии — там app extension загружается автоматически).

### Notarization (для распространения)

Для запуска на чужих Mac приложение нужно подписать Developer ID сертификатом,
нотаризовать (`xcrun notarytool`) и застеплить (`xcrun stapler staple`). Системные
расширения требуют подписи Developer ID и не запустятся из непроверенного источника.

### Mac App Store вместо Developer ID (альтернатива)

Если распространять через Mac App Store, packet tunnel можно упаковать как
**app extension** (как в iOS). Тогда:
- в `project.yml` смените тип таргета `PacketTunnel` c `system-extension` на `app-extension`;
- в обоих entitlements смените `packet-tunnel-provider-systemextension` на `packet-tunnel-provider` и уберите `com.apple.developer.system-extension.install` из entitlements приложения;
- App Group можно оставить в форме `group.*` (для App Store сборок это допустимо);
- удалите `SystemExtensionManager.swift` и вызовы `sysext.*` в `ContentView.swift` (шаг активации не нужен).

## Ответы на частые вопросы, решение проблем

- **Q**: подключение установилось, выход в Интернет есть, но медленно.
- **A**: скорее всего, это Legacy режим (из-за шейпинга VK ~1.5-2 Мбит/с на 30 соединениях). Либо маленькое число соединений в настройках. Теоретический максимум одного соединения (без шейпинга) — ~2 Мбит/с, 30 соединений в идеале дают 50+ Мбит/с.

- **Q**: подключение установилось, но выхода в Интернет нет.
- **A**: проверьте `tcpdump` на сервере, общается ли серверная часть vk-turn-proxy с WireGuard сервером. Если пакеты идут только в сторону WireGuard, но обратно нет — вероятно, несоответствие ключей/адресов WireGuard между клиентом и сервером, либо WireGuard не слушает на нужном IP:порт. Если пакеты ходят в обе стороны, а Интернета нет — проблема дальше (NAT/firewall/маршрутизация на сервере).

- **Q**: приложение просит разрешить системное расширение при каждом обновлении.
- **A**: это нормально — при смене подписи/версии macOS может заново запросить одобрение расширения. Разрешите его один раз в Системных настройках.
