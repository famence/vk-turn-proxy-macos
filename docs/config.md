# Настройка config.json — что откуда брать и куда вписывать

Это пошаговая инструкция по заполнению `config.json` для `vk-turn-socks`
(лёгкий SOCKS5/HTTP-вариант и менюбар-агент). Если что-то из полей непонятно —
здесь расписано, откуда взять каждое значение.

## Коротко: как это устроено

Сервис на вашем Mac строит WireGuard-туннель до **вашего сервера** (VPS) поверх
VK TURN-релея. Значит, где-то уже должен быть настроен сервер:

- VPS с публичным IP, на нём запущено **серверное** приложение `vk-turn-proxy`
  (то, что слушает `-listen IP:порт`);
- на том же VPS поднят **WireGuard-сервер**, к которому подключается наш
  встроенный WireGuard-клиент.

`config.json` описывает именно **клиентскую** сторону: какой VK-звонок
использовать, куда стучаться (сервер) и какими WireGuard-ключами.

> Если вы уже пользовались iOS-приложением — сервер у вас уже есть, и все ключи
> тоже уже есть. Проще всего перенести их (раздел «Способ A»).

---

## Способ A (самый простой): перенести настройки из iOS-приложения

В iOS-приложении есть «1-Click Connection Link» — строчка вида
`vkturnproxy://import?data=…`, в которой закодированы все настройки (ключи,
адрес сервера, ссылка на звонок, режим). `vk-turn-socks` умеет превратить такую
ссылку сразу в готовый `config.json`.

Где взять ссылку:

- **из скрипта** `quick_link.py` (лежит в этом репозитории): заполните в нём
  `CONFIG` своими значениями (или сгенерируйте новый ключ, см. Способ B) и
  запустите — он напечатает `vkturnproxy://…`;
- **или** попросите ссылку у того, кто настраивал вам сервер (её обычно и
  раздают для быстрой настройки);
- **или** сделайте в iOS-приложении «Export Full Backup» (получится `.json`
  файл) — его тоже можно импортировать.

Импорт в готовый конфиг:

```shell
# из ссылки:
./vk-turn-socks-darwin-arm64 -import 'vkturnproxy://import?data=…' -config config.json

# или из файла бэкапа:
./vk-turn-socks-darwin-arm64 -import ./vkturnproxy-backup-….json -config config.json
```

Команда создаст `config.json` со всеми полями. После этого сразу запускайте
(`-config config.json`). Если в выводе есть WARNING про пустой
`private_key`/`peer_public_key` — значит в ссылке их не было (например, ссылка
для режима SRTP-WRAP-A их не содержит, это нормально); иначе допишите вручную.

## Способ B (с нуля): сгенерировать WireGuard-ключи

`private_key` — это **приватный ключ WireGuard-клиента**. Он не «где-то лежит» —
его создают. У WireGuard всегда пара: приватный ключ (остаётся на клиенте) и
парный ему публичный ключ (кладётся на сервер в раздел `[Peer]`).

Способ B1 — прилагаемым скриптом (ничего ставить не нужно, только Python 3):

```shell
python3 quick_link.py -gen-peer-key 192.168.102.7/24
```

Скрипт напечатает две вещи:

1. `vkturnproxy://import?data=…` — ссылку с уже вшитым **новым приватным
   ключом**. Скормите её `vk-turn-socks -import …` (см. Способ A) — и `config.json`
   готов.
2. Блок `[Peer]` — его нужно добавить в конфиг **WireGuard-сервера** на VPS
   (там уже есть публичный ключ, соответствующий вашему новому приватному) и
   перезапустить WireGuard. `192.168.102.7/24` — адрес клиента в туннеле; для
   каждого клиента берите свой (`.7`, `.8`, …).

   Перед запуском в `quick_link.py` в словаре `CONFIG` заполните `peerPublicKey`,
   `vkLink`, `peerAddress` — они одинаковы для всех клиентов (это про ваш
   сервер), меняется только ключ+адрес клиента.

Способ B2 — стандартными утилитами WireGuard (если установлен `wg`):

```shell
wg genkey | tee client_private.key | wg pubkey > client_public.key
cat client_private.key   # → это ваш wireguard.private_key
cat client_public.key    # → это добавить в [Peer] на сервере (PublicKey = …)
```

`peer_public_key` в конфиге — это **публичный ключ сервера** (строка `PublicKey`
из раздела `[Interface]` серверного WireGuard, или то, что вам дал админ).

## Поле за полем

Верхний уровень:

| Поле | Что это | Откуда взять |
|------|---------|--------------|
| `vk_link` | Ссылка на VK-звонок | Создать звонок в VK и скопировать ссылку `https://vk.ru/call/join/…` (или `https://vk.me/join/…`). Звонок должен быть активен. |
| `peer_addr` | Адрес **вашего** сервера `vk-turn-proxy` | `IP:порт`, то, что на сервере задано в `-listen` (напр. `203.0.113.10:56000`). НЕ адрес TURN-релея. |
| `mode` | Режим сервера | `srtp` (по умолчанию/рекомендуется), `legacy`, `srtp-wrap`, `srtp-wrap-a`. Должен совпадать с тем, как запущен сервер. |
| `use_udp` | Транспорт клиент↔TURN | `false` (TCP, устойчивее и обычно быстрее из-за особенностей VK) — оставьте так, если не уверены. |
| `num_conns` | Число соединений | `30` — баланс скорости/нагрузки. Больше соединений → выше скорость, но и больше запросов к VK. |
| `turn_server`/`turn_port` | Необязательный форс конкретного TURN-релея | Обычно пусто (используется релей из ссылки звонка). |
| `wrap_key_hex` | Ключ для `srtp-wrap` | 64 hex-символа, совпадает с `-wrap-key` на сервере. Только для `mode: srtp-wrap`. |
| `wrap_a_password` | Пароль для `srtp-wrap-a` | «Главный пароль туннеля» сервера WDTT. Только для `mode: srtp-wrap-a` (тогда блок `wireguard` не нужен). |
| `socks_listen` | Где слушать SOCKS5 | `127.0.0.1:1080` — это адрес, который укажете в Surge. |
| `http_listen` | Где слушать HTTP-прокси | `127.0.0.1:1087` (необязательно; пусто — выключено). |
| `cookie_header` | Вход по VK-аккаунту | Необязательно. `remixsid=…; p=…` из залогиненной сессии VK — тогда без анонимного входа и без капчи. |

Блок `wireguard` (не нужен только в режиме `srtp-wrap-a`):

| Поле | Что это | Откуда взять |
|------|---------|--------------|
| `private_key` | Приватный ключ **клиента** (base64) | Сгенерировать (Способ B) или перенести из iOS (Способ A). |
| `peer_public_key` | Публичный ключ **сервера** (base64) | `PublicKey` из `[Interface]` серверного WireGuard / от админа. |
| `preshared_key` | Доп. ключ (base64), необязательно | Только если на сервере в `[Peer]` задан `PresharedKey`; иначе оставьте `""`. |
| `address` | IP клиента в туннеле | Должен совпадать с `AllowedIPs` этого клиента в `[Peer]` на сервере, напр. `192.168.102.7/24`. |
| `dns` | DNS через туннель | `1.1.1.1` (или свой). Резолвинг идёт через туннель — без утечек. |
| `mtu` | MTU туннеля | `1280` (безопасное значение). |
| `persistent_keepalive` | Keepalive, сек | `25`. |

## Готовый пример (режим SRTP)

```json
{
  "vk_link": "https://vk.ru/call/join/abcdefghijklmnop",
  "peer_addr": "203.0.113.10:56000",
  "mode": "srtp",
  "use_udp": false,
  "num_conns": 30,
  "wireguard": {
    "private_key": "cGhvbnktcHJpdmF0ZS1rZXktMzJieXRlcy1iYXNlNjQ=",
    "peer_public_key": "c2VydmVyLXB1YmxpYy1rZXktMzJieXRlcy1iYXNlNjQ=",
    "preshared_key": "",
    "address": "192.168.102.7/24",
    "dns": "1.1.1.1",
    "mtu": 1280,
    "persistent_keepalive": 25
  },
  "socks_listen": "127.0.0.1:1080",
  "http_listen": "127.0.0.1:1087"
}
```

Запуск и подключение из Surge — в [docs/socks.md](socks.md).

## Частые вопросы

- **`srtp handshake: dtls client handshake: context deadline exceeded` (bootstrap не проходит, интернета нет)** —
  это САМАЯ частая проблема. Значит: вход в VK и аллокация TURN-релея прошли
  успешно, но **DTLS/SRTP-рукопожатие с вашим сервером через релей не
  завершилось**. Пакеты доходят до релея VK, а ваш сервер за ним не отвечает.
  Это проблема сервера/`peer_addr`/`mode`, а не Surge. Проверьте по порядку:
  1. **`mode` должен совпадать с тем, как запущен сервер.** `mode: srtp` требует
     сервера, запущенного с `-srtp`. Если сервер работает в legacy/WRAP/WRAP-A
     или SRTP слушает на другом порту — рукопожатие падает ровно так.
  2. **`peer_addr`** — это `IP:порт` `-listen` вашего `vk-turn-proxy` (именно
     SRTP-слушателя), НЕ порт WireGuard и НЕ адрес TURN-релея.
  3. Сервер должен быть запущен и **достижим со стороны релея VK** (на firewall
     открыт входящий доступ к этому IP:порт).
  4. Раз iOS-приложение работает с тем же сервером — надёжнее всего скопировать
     его настройки один в один: сделайте в приложении connection-link (или
     Export Full Backup) и выполните
     `./vk-turn-socks -import '<ссылка-или-файл>' -config config.json`.
     Скорее всего, при ручном заполнении не совпал `mode` или `peer_addr`.

- **Туннель «up», но интернета нет; в логе `SRTP read timeout, tunnel stale … reconnecting` каждые ~30 с, `tx` растёт, а `rx=0`** —
  транспорт до сервера уже работает (`SRTP+TURN session established`), но
  **WireGuard-рукопожатие не получает ответа**. Клиент шлёт handshake каждые
  ~5 с (по 148 байт — отсюда рост `tx`), сервер молчит (`rx=0`), watchdog
  считает туннель мёртвым и переподключается по кругу. Почти всегда это
  **несовпадение WireGuard-ключей/адреса** (сервер молча отбрасывает
  рукопожатие от незнакомого публичного ключа):
  1. `wireguard.peer_public_key` — это публичный ключ **сервера** WireGuard.
  2. **Публичный** ключ вашего `wireguard.private_key` должен быть добавлен как
     `[Peer]` на WireGuard-сервере, а его `AllowedIPs` — покрывать
     `wireguard.address` (ваш IP в туннеле).
  3. Если для Mac вы сгенерировали **новый** ключ (`quick_link.py -gen-peer-key`),
     то напечатанный блок `[Peer]` нужно добавить на сервер и перезапустить
     WireGuard — иначе сервер не отвечает.
  4. Не запускайте iOS и Mac **одновременно с одинаковым ключом и адресом** —
     они будут «драться» за один peer. Дайте каждому устройству свой ключ и
     свой IP в туннеле (`.7`, `.8`, …), каждый со своим `[Peer]` на сервере.
  5. Надёжнее всего — `-import` точные рабочие настройки из iOS (там ключ уже
     зарегистрирован на сервере), но учтите п.4 про одновременную работу.
  Сама утилита теперь печатает такое предупреждение автоматически через ~25 с,
  если `rx` остаётся нулевым.

- **«Не помню приватный ключ»** — приватный ключ восстановить нельзя, он есть
  только у клиента. Если он потерян: сгенерируйте новый (Способ B), добавьте
  новый `[Peer]` (с новым публичным ключом) на сервер — старый можно удалить.
- **«Подключается, но нет интернета»** — обычно несоответствие WireGuard между
  клиентом и сервером: проверьте, что `peer_public_key` = публичный ключ
  сервера, а публичный ключ ВАШЕГО `private_key` добавлен в `[Peer]` на сервере,
  и `address` совпадает с `AllowedIPs` там же.
- **«Где взять сам сервер»** — это отдельная серверная часть `vk-turn-proxy`
  (бинарники в [Releases оригинального проекта](https://github.com/anton48/vk-turn-proxy/releases))
  + WireGuard-сервер на VPS. Настройка сервера описана в [docs/setup.md](setup.md).
- **`mode` какой выбрать** — тот же, что на сервере. Если сервер запущен с
  `-srtp` → `srtp`; без флагов → `legacy`; WDTT → `srtp-wrap-a`.
